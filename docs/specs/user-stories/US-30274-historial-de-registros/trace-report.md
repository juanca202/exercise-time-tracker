## Resumen

AC-001, AC-002, AC-003, AC-004 y AC-006 (lectura del historial, totales por Tarea/Proyecto/mes con navegación, y resumen del periodo) están completamente cubiertos por pruebas unit/componente y e2e, todas ejecutadas y en verde (9 archivos/21 tests Vitest + 3 e2e Playwright). Dos criterios quedan **Parciales**: AC-005 (listado de campos) cubre el caso general (TC-010) pero no el límite explícito de duración mínima de 1 minuto (TC-011); y AC-007 (carga en menos de 2s hasta 1000 registros) solo está probado a nivel de las funciones de agregación puras (`selectors.performance.test.ts`) — ningún artefacto mide el tiempo real de navegación + render de `TimeHistoryScreen` con volumen (ni 50 ni 1000 registros), que es literalmente lo que exigen TC-014 y TC-015.

## Observaciones y pendientes

- **AC-007 (TC-014/TC-015):** construir un harness e2e de rendimiento equivalente al de TK-30304 (US-30272) — sembrar 50 y 1000 Registros de Tiempo (vía fixture/seeding en `localStorage`, dado el costo de generarlos por UI) y medir con `performance.now()` el tiempo desde la navegación a `/history` hasta que el historial completo (tabla + resúmenes) está visible, con margen de tolerancia documentado para CI. Es la brecha más relevante: hoy solo está probado que el cálculo es rápido, no que la pantalla se cargue y pinte dentro del umbral del AC.
- **AC-005 (TC-011):** agregar un caso (unit o componente) con un Registro de exactamente 1 minuto de Duración para confirmar que se renderiza sin errores de formato (`00:01:00`), cerrando el caso límite explícito del TC.
- **AC-004 (TC-008 e2e):** el escenario e2e de navegación entre meses no siembra datos por mes ni verifica que el contenido cambie, solo la etiqueta — no bloquea el veredicto porque el recálculo real ya está probado a nivel unit, pero conviene reforzarlo si se busca cobertura e2e completa del AC.
